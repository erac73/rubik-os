The Rubik's Cube Model

El Modelo del Cubo Rubik

Rubik OS organizes the operating system as a Rubik's Cube: 6 faces, each with 9 cells (F0-F4) or 11 cells (F3), totaling 58 atomic cells.

Rubik OS organiza el sistema operativo como un Cubo Rubik: 6 caras, cada una con 9 celdas (F0-F4) u 11 celdas (F3), totalizando 58 celdas atómicas.

Architectural Principles

Principios Arquitectónicos

1. Decentralization

1. Descentralización

Each cell is an independent process/service. If a cell fails, the rest continue operating. Cells communicate via lightweight IPC (D-Bus, Unix sockets, shared memory).

Cada celda es un proceso/servicio independiente. Si una celda falla, las demás continúan funcionando. Las celdas se comunican mediante IPC ligero (D-Bus, sockets Unix, memoria compartida).

2. Atomicity

2. Atomicidad

Each cell does exactly one thing. A "swap management" cell knows nothing about networking. A "firewall" cell does not manage files.

Cada celda hace exactamente una cosa. Una celda de "gestión de swap" no sabe de redes. Una celda de "firewall" no gestiona archivos.

3. Hot-swappable (Rotation)

3. Intercambiabilidad (Rotación)

Any cell can be replaced at runtime. Don't like the default package manager? Swap that cell without touching the rest.

Cada celda puede ser reemplazada en caliente. ¿No te gusta el gestor de paquetes por defecto? Cambia esa celda sin tocar el resto.

4. Memory Efficiency

4. Eficiencia de Memoria

Aggressive ZRAM compression, kernel tuned for low latency and low memory, on-demand services (socket-activation, timer-activation), shared memory pool between face cells.

Uso agresivo de ZRAM, kernel configurado para baja latencia y menor uso de memoria, servicios bajo demanda (socket-activation, timer-activation), pool de memoria compartida entre celdas de la misma cara.

5. Flat Hierarchy

5. Jerarquía plana

No unnecessary abstraction layers. Each cell communicates directly with whoever it needs. No layered architectures that add latency.

No hay capas de abstracción innecesarias. Cada celda se comunica directamente con quien necesita. Sin arquitecturas en capas que agreguen latencia.

The Rubik Orchestrator (rubikd)

El Orquestador Rubik (rubikd)

The central orchestrator manages cell startup/shutdown, dependencies (acyclic graph), rotation (hot updates), health checks, and memory metrics collection.

El orquestador central gestiona inicio/parada de celdas, dependencias (grafo acíclico), rotación (actualización en caliente), health checks y recogida de métricas de memoria.

rubikctl status          # Status of all cells# Estado de todas las celdas
rubikctl rotate face-0   # Replace all cells in a face# Reemplazar todas las celdas de una cara
rubikctl cell enable F0.4 memory-compression
rubikctl cell disable F2.2 defrag

Cell Lifecycle

Ciclo de Vida de una Celda

REGISTERED → LOADED → ACTIVE → HEALTHY
                          ↓
                     DEGRADED → RETRY → ACTIVE
                          ↓
                      FAILED → NOTIFY → RESTART

The orchestrator monitors each cell's state and executes corrective actions automatically.

El orquestador monitorea el estado de cada celda y ejecuta acciones correctivas automáticamente.
